## Теория: что такое API-тестирование и зачем оно нужно

API-тестирование проверяет серверную логику без UI: мы отправляем HTTP-запросы к endpoint'ам и проверяем:

1. Статусы ответов (`200`, `201`, `400`, `403`, `404`).
2. Структуру и содержимое JSON.
3. Бизнес-правила (RBAC, валидация, scope доступа).
4. Негативные сценарии (ошибочные данные, запрет доступа, отсутствующие сущности).

В этом наборе используются **интеграционные API-тесты**: они обращаются к живому серверу, а не мокам.

Пример из файла тестов:

```python
response = session.get(f"{base_url}/api/health", timeout=TIMEOUT)
assert response.status_code == 200
payload = response.json()
assert payload.get("status") == "ok"
```

Здесь сразу проверяются и транспортный уровень (код ответа), и бизнес-смысл (`status == "ok"`).

## Какие фреймворки, библиотеки и API использованы

### 1) `pytest` (тестовый фреймворк)

Используется для:
- организации тестов по функциям/классам,
- фикстур (`@pytest.fixture`),
- проверок через `assert`,
- пропуска всего набора, если сервер недоступен (`pytest.skip(...)`).

Пример фикстуры с областью жизни `session`:

```python
@pytest.fixture(scope="session")
def session():
    s = requests.Session()
    yield s
    s.close()
```

`yield` в фикстуре делит ее на setup/teardown: до `yield` создается сессия, после — закрывается.

### 2) `requests` (HTTP-клиент)

Используется для отправки запросов:
- `session.get(...)`
- `session.post(...)`
- `session.delete(...)`

Ключевые параметры:
- `json=...` — отправка JSON-тела,
- `headers=...` — передача заголовков авторизации/роли,
- `timeout=TIMEOUT` — ограничение времени ожидания ответа.

### 3) Тестируемое API

По тестам видно, что проверяются endpoint'ы:
- `GET /api/health`
- `GET /api`
- `POST /api/auth/register`
- `GET /api/users`
- `POST /api/clients`
- `GET /api/clients/{id}`
- `DELETE /api/clients/{id}`

- Запрос на `/api/health`.
- Ожидается `200`.
- В теле: `{"status": "ok"}`.

Дополнительно есть автопроверка в фикстуре `require_live_server`, которая пропускает весь набор тестов, если сервер не доступен или health-check не `200`.

### 2) Проверка документации/мета-информации API (`GET /api`)

**Проверка:** `test_api_root_contains_docs_and_endpoints`
- Ожидается `200`.
- Проверяются ключи `name`, `health`, `httpBase`, `auth`.
- Проверяется описание auth endpoint'ов:
  - `POST /api/auth/login`
  - `GET /api/auth/me`

Это проверка API-контракта на корневом информационном endpoint.

### 3) Негативный сценарий: отключенная регистрация

**Проверка:** `test_auth_register_is_disabled`
- `POST /api/auth/register`.
- Ожидается `403`.
- Проверяется текст ошибки: содержит `"Регистрация отключена"`.

Это пример проверки бизнес-ограничения на уровне API.

### 4) Разграничение доступа (RBAC): менеджер не может читать admin-only ресурс users

**Проверка:** `test_manager_cannot_access_users_resource`
- `GET /api/users` с заголовками менеджера (`HEADERS_MANAGER_IVANOV`).
- Ожидается `403`.
- Сообщение должно указывать на необходимость прав администратора.

Это тест ролевой модели доступа.

### 5) Валидация входных данных: создание клиента без обязательного `name`

**Проверка:** `test_create_client_without_required_name_fails_validation`
- `POST /api/clients` с payload без `name`.
- Ожидается `400`.
- В ответе должно быть поле `message`.

Это тест серверной валидации DTO/схемы запроса.

### 6) Неверный ID: `400 Invalid id`

**Проверка:** `test_get_client_with_invalid_id_returns_400`
- Запрос к `/api/clients/not-a-number`.
- Ожидается `400`.
- Проверяется точный текст: `"Invalid id."`.

Проверяется ветка обработки синтаксически невалидного идентификатора.

### 7) Отсутствующая сущность: `404 Client not found`

**Проверка:** `test_get_missing_client_returns_404`
- Запрос к явно несуществующему id `/api/clients/999999999`.
- Ожидается `404`.
- Сообщение: `"Client not found."`.

Отличие от предыдущего кейса:
- `400` — формат id неверный,
- `404` — формат корректный, но сущность не найдена.

### 8) Сквозной сценарий менеджера: create/read + scope + cleanup

**Проверка:** `test_manager_creates_client_in_own_scope`
- Создание клиента менеджером (`201`).
- Проверка, что `managerId` выставляется сервером из авторизации:
  `created.get("managerId") == int(HEADERS_MANAGER_IVANOV["x-user-id"])`.
- Владелец может читать клиента (`200`).
- Другой менеджер не может читать чужого клиента (`404`).
- В `finally` выполняется cleanup через `DELETE` от администратора.

Это классический end-to-end сценарий с обязательной очисткой тестовых данных.

## Объяснение синтаксических конструкций из файла

### `@pytest.fixture(..., autouse=True)`

Автоматическая фикстура запускается без явной передачи в тест-функцию.
В файле это использовано для:
- предварительной проверки живого сервера (`require_live_server`),
- печати имени теста перед выполнением (`print_test_caption`).

### `scope="session"`

Означает, что фикстура создается один раз на всю сессию тестирования. Это ускоряет тесты и уменьшает количество повторных подключений.

### `try/finally` в тесте

Используется, чтобы cleanup происходил всегда, даже если одна из проверок упадет:

```python
created_client_id = None
try:
    # создание и проверки
finally:
    if created_client_id:
        session.delete(...)
```

### Генерация уникальных данных

```python
suffix = uuid.uuid4().hex[:8]
```

Позволяет избегать конфликтов данных между прогонами тестов (уникальные `name`/`email`).

### Проверки через `assert`

В `pytest` используется обычный `assert`, но фреймворк дает подробный отчет при падении:
- сравнение фактического и ожидаемого значений,
- трассировка места ошибки.

## Почему этот подход к тестам правильный

- Покрыты и позитивные, и негативные сценарии.
- Проверяется безопасность (RBAC).
- Проверяется валидация входа.
- Разделены ошибки типа `400` и `404`.
- Есть жизненно важный cleanup тестовых данных.
- Есть precondition-проверка доступности сервера, чтобы не получать ложные падения.
