## 1. Зачем нужен CI/CD

**CI (Continuous Integration)** — непрерывная интеграция: при каждом изменении кода автоматически собирается проект, запускаются проверки (линтеры, юнит- и интеграционные тесты), артефакты фиксируются. Цель — как можно раньше узнать, что ветка «сломана», до того как изменения попадут к остальной команде.

**CD** обычно расшифровывают как **Continuous Delivery** (непрерывная поставка) или **Continuous Deployment** (непрерывное развёртывание): после успешного CI готовая версия может автоматически выкатываться на стенд или в продакшен по согласованным правилам.

**CI/CD** — связка практик и инструментов: автоматизация сборки, тестов, проверки качества и доставки изменений.

## 2. Теория: GitHub Actions (основной вариант лабораторной)

GitHub Actions — встроенный в GitHub движок автоматизации. Конфигурация хранится в репозитории в каталоге **`.github/workflows/`**, обычно в файлах с расширением **`.yml`**.

Основные понятия:

| Понятие | Смысл |
|--------|--------|
| **Workflow** | Описанный процесс: когда запускается и из каких шагов состоит |
| **Event (триггер)** | Например `push`, `pull_request`, расписание `schedule` |
| **Job** | Набор шагов, выполняемых на одном раннере; несколько job могут идти параллельно или с зависимостями |
| **Step** | Отдельная команда или действие (`uses`, `run`) |

Типичный сценарий для учебного проекта:

1. Событие **`push`** (и при необходимости **`pull_request`**) в выбранные ветки.
2. Job **checkout** репозитория, установка окружения (например Python/Node).
3. Установка зависимостей и запуск **pytest** и других автотестов.
4. (Опционально) публикация отчётов или артефактов.

В лабораторной работе требуется завести workflow (в задании — файл вроде **`ci.yml`**) и убедиться, что после отправки кода на GitHub пайплайн **запускается** и **виден** в интерфейсе Actions.

## 3. Ветвление: `main`, `dev`, `fix`

Рекомендуемая модель из методички:

- **`main`** — стабильная линия, пригодная для релиза/демонстрации (в т.ч. для CD).
- **`dev`** — интеграционная ветка разработки.
- **`fix`** (или ветки с префиксом `fix/…`) — исправления дефектов; обычно ответвляются от актуальной **`dev`**, затем сливаются обратно в **`dev`** через pull request.

Прямые коммиты в **`main`** в учебном сценарии заменяются на **merge через Pull Request** после ревью.

## 4. Практика CI (командный сценарий)

Отработка типового цикла:

1. Клонировать репозиторий, перейти на ветку **`fix`** (или создать **`fix/…`** от **`dev`**).
2. Внести осознанную «поломку» или дефект, закоммитить и отправить на GitHub.
3. Исправить код в той же ветке, снова закоммитить и отправить.
4. Создать **Pull Request** из **`fix`** в **`dev`**, провести **code review**, дождаться прохождения **CI** на PR.
5. Выполнить **merge** в **`dev`**.
6. Создать второй **Pull Request** из **`dev`** в **`main`**, снова проверить CI и выполнить merge.

Так моделируется разделение ответственности, обязательные проверки до попадания кода в стабильную ветку и прозрачная история изменений.

## 5. Практика CD (пример с GitHub Pages)

**CD** в задании может включать настройку **автоматической публикации** (например, статики фронтенда или отчёта) на **GitHub Pages** так, чтобы **актуальная версия** для внешнего просмотра обновлялась **после слияния в `main`** (или по выбранной политике — важно зафиксировать правило и следовать ему).

## 6. Альтернатива в методичке: Jenkins

Второй блок теории посвящён **Jenkins**: отдельный сервер CI, job-ы, плагины, агенты. Смысл тот же, что у GitHub Actions (автоматизация шагов по событию), но конфигурация и инфраструктура другие. Для выполнения лабораторной по варианту с Jenkins ориентируйтесь на официальные туториалы по ссылке выше и материалы курса в Moodle.
