# Лабораторная 9: Юнит-тестирование (теория + практика по проекту)

## 1) Что такое юнит-тестирование

Юнит-тестирование — это проверка отдельных небольших частей программы (юнитов): функций, методов, модулей, сервисов.  
Цель — убедиться, что каждый юнит работает корректно изолированно, а ошибки выявляются как можно раньше.

В рамках этой лабораторной важно:
- писать 10-15 тестов;
- использовать разные входные данные;
- проверять сценарии ошибок и исключений;
- изолировать зависимости (через тестовые заглушки / моки);
- оценить покрытие кода.


## 2) Кто и когда проводит юнит-тестирование

- Обычно разработчик пишет и запускает юнит-тесты во время реализации функции и перед коммитом.
- Также тесты автоматически запускаются в CI/CD (например, при push или pull request).


## 3) Преимущества и недостатки

### Преимущества
- Раннее обнаружение ошибок.
- Безопасный рефакторинг (быстрая проверка, что поведение не сломано).
- Документация поведения кода через тест-кейсы.
- Снижение стоимости исправления дефектов.

### Недостатки
- Требуют времени на написание и поддержку.
- Плохо спроектированный код трудно тестировать.
- Юнит-тесты не заменяют интеграционные и e2e-тесты.


## 4) Структура юнит-теста (AAA)

Классическая структура:
- **Arrange** — подготовка данных и окружения.
- **Act** — вызов тестируемого действия.
- **Assert** — проверка результата.

Пример из проекта:
- Arrange: подготовка `session_stub` и ответа от API;
- Act: вызов функции `fetch_users(...)`;
- Assert: проверка, что был вызван нужный endpoint и вернулись ожидаемые данные.

См. файл: [`tests/tests_server.py`](./tests_server.py)


## 5) Зачем изолировать юниты

Изоляция нужна, чтобы тест падал только из-за ошибки в проверяемом юните, а не из-за внешних факторов:
- недоступна БД;
- упал HTTP-сервис;
- плохая сеть;
- изменились данные в живой базе.

В проекте изоляция показана через `Mock(spec=requests.Session)`:
- нет реального HTTP-вызова;
- ответы задаются прямо в тесте;
- можно надежно воспроизводить сценарии успеха и ошибок.

См. файл: [`tests/tests_server.py`](./tests_server.py)


## 6) Тестовые двойники

Основные виды:
- **Dummy** — передается как "заглушка", но не используется.
- **Stub** — возвращает заранее заданные данные.
- **Spy** — записывает, как его вызывали.
- **Mock** — и имитирует поведение, и проверяет ожидания вызовов.
- **Fake** — рабочая, но упрощенная реализация (например, in-memory хранилище).

В данной лабораторной используется в основном **Mock/Stub** для HTTP-сессии.


## 7) Виды покрытия тестами

Часто применяются:
- покрытие строк (Line Coverage);
- покрытие ветвлений (Branch Coverage);
- покрытие функций/методов;
- покрытие условий (Condition Coverage).

Минимальная практическая метрика — покрытие строк.  
Для Python можно использовать `pytest-cov`.

Пример команды:
```bash
pytest tests/tests_server.py --cov=. --cov-report=term-missing
```


## 8) Практика по этому проекту

В проекте есть два уровня тестирования:

1. **Интеграционные HTTP-тесты** (проверяют живой сервер и БД):
   - файл: [`tests/test_server.py`](./test_server.py)
   - используют реальные HTTP-запросы (`requests`) к API.

2. **Юнит-тесты с изоляцией зависимостей**:
   - файл: [`tests/tests_server.py`](./tests_server.py)
   - используют моки вместо реального API.

### Что проверяется в `tests/tests_server.py`

- Успешная обработка ответа (`200`).
- Обработка ошибок (`400/404/500`) с выбросом исключения.
- Поведение при невалидном JSON в ответе.
- Проброс сетевого исключения (например, `requests.Timeout`).
- Корректность вызова endpoint и передаваемого payload.
- Пустые данные как валидный сценарий (пустой список пользователей).

Это соответствует требованиям лабораторной: разные тестовые данные, сценарии ошибок, проверка исключений и изоляция.
